三十天自制操作系统day3-进入32位模式并导入C语言

摘要

本文主要介绍《30天自制操作系统》一书第三天的内容。包括：如何使用引导扇区进行操作系统的加载以及导入C语言。从内容上看，第三天的内容可以大致分为两部分，第一部分是修改ipl.nas, 使其能够满足加载指定数目的磁盘内容(最后实现上是完成了10个扇面的加载)；第二部分是完成了一个简单的C语言编写的“操作系统”，其功能是实现不断的空转。

IPL加载磁盘内容

第二天完成的ipl.nas程序，虽然名为ipl(init program loader)，但是并没有真正加载任何程序。今天我们就会解决这个问题，修改ipl.nas, 让它能真正读取磁盘内容并装载到内存中。

书中关于加载磁盘这部分，使用了4个样例项目，分别是harib00a到harib00d, 它们的区别如下：

1. harib00a: 使用简单的BIOS中断读取一个扇区的磁盘内容
 2. harib00b: 在harib00a的基础上，增加了重读5次的功能
 3. harib00c: 一次性读取18个扇区的内容
 4. harib00d: 一次性读取10个扇面的内容

依次将这些内容呈现出来，会使得本文篇幅过长，所以这里仅仅只介绍harib00a和harib00d的实现，其中harib00d包含了中间两个项目的内容。

harib00a: 简单读取一个扇区

ipl.nas最终实现

其中大部分内容都和昨天的ipl.nas没有什么区别，区别在于我们从entry进入后，不是直接打印信息，而是装载磁盘了。当我们设定好磁盘BIOS的各种参数之后，使用一个INT 0x13指令，调用磁盘读取例程，就会将内容磁盘指定位置的内容(由扇面，磁头，扇区决定)装载到指定的内存位置(由[ES:BX]寄存器的值决定)。如果失败，FLAGS.CF位会被置1，使用JC指令，表示，一旦有进位(即FLAGS.CF位为1)，就跳转到错误位置，打印报错信息。

关于如何读写磁盘，下列网站有详细的规定：https://stanislavs.org/helppc/int_13.html。这里截下来一小部分简要说明以下：

可以看出，这里有多种磁盘BIOS例程，但它们都使用INT 0x13指令来触发，通过设置一些寄存器不同的值来区分这些不同的例程。在今天的内容中，暂时只会用到 INT 13,2 和 INT 13, 0，即读取磁盘和重置磁盘状态。关于其详细解读，如下(以 INT 13,2 为例)：

这里显示了如果我们需要调用读取磁盘例程时要设置的寄存器的值：

​ AH = 02 : 表示是读磁盘操作，顺带一提，写磁盘是0x03, 校验是0x04, 寻道是0x0c

​ AL: 要读取的扇区的数目，只能同时处理连续的扇区

​ CH: 柱面号

​ CL: 扇区号

​ DH: 磁头号

​ DL: 磁盘驱动器号

​ [ES:BX]: 两个寄存器共同组成了加载到的内存地址。

在了解上述规定之后，就不难理解ipl.nas中的写法了。我们开始分别将0,0,2写入CH, DH, CL寄存器的目的就是分别将读取的柱面，磁头，扇区数设置为2，告诉BIOS，我们将从第0柱面的第二个扇区读取，读取的磁头号是0。这样设置的原因在于，启动扇区位于磁盘的第一个扇区，后面紧接着的就是操作系统内核，注意这里的扇区数是设置为2，而不是1，是因为扇区数是从1开始计数，而不是像通常的0那样开始计数的。关于柱面，磁头，扇区是什么，只要对存储系统有所了解应该都不陌生，这里就不详细介绍了，网上有大量关于这方面的介绍。

接下来我们使用MOV AH, 0x02将AH寄存器设置为2，表示是要读取磁盘。将AL设置为1，表示读取一个扇区。设置DL为0，表示调用A驱动器。

值得注意的是[ES: BX]寄存器的含义，在 INT 13,2 例程中，它表示装载的目的内存地址， 那为什么不直接使用一个寄存器的值作为内存地址呢？原因在于一个寄存器只有16位，至多能索引64K的地址，使用一个寄存器对表示地址，能提升地址空间范围。但是这里的地址计算并不是指将两个寄存器的值拼接在一起就表示一个地址，正确的地址计算公式如下：

就是说，是将ES寄存器中的值左移4位，再加上BX中的值，这样寻址范围可扩大到1MB。这样，我们就不难理解代码中的设置的含义了。先使用MOV AX,0x0820和 MOV ES, AX将 EX的值设置为0x0820, 后面又将BX的值设置为0， 这就表明了目的加载地址是0x8200。

最后我们要谈论的是JC error这条指令，JC的意思是Jump if Carry，即如果有进位就跳转。这里的进位是指FLAGS寄存器的CF表示位，正如说明中提到的那样，CF为1表示读取失败，JC就会发生跳转，到error的位置，进行错误信息的打印。不过正常的处理应当是一次读取失败后继续尝试，当指定次数重试仍失败时才打印报错信息，这也是我们接下来要解决的问题。

harib00d: 连续读取10个柱面的内容

ipl10.nas新增内容

上述内容可以直接增加到entry 设定CH,DH,CL等寄存器之后的部分。增加的部分有3个标签：readloop, retry和next，我们分别对这三个部分进行讲解。

readloop和retry

readloop包含了一个完整的读取一个扇区的内容的代码，内容包括第一句MOV SI, 0和标签retry的所有语句。这里简要说明一下：在上文中我们已经提到，有的时候读取磁盘会失败，通常我们的处理就是再试一次，如果重试次数超过5，就放弃并打印报错信息。我们用SI寄存器来放置重试的次数，每当我们读取一个新的扇区时，就将SI设置为0，然后进行接下来的操作。具体读取和重试的代码在retry中。

在retry中，我们先设定了寄存器AH,AL,BX,DL的值，然后通过INT 0x13指令调用BIOS例程。使用JNC next查看FLAGS.CF位，如果成功读取，就跳转到next, 为读取下一个扇区做准备，否则读取该扇区失败，顺着控制流向下，我们递增SI的值，表示重试次数增加了，并与5比较，如果SI的值大于等于5(说明至少5次重试了)，就跳转到error位置打印报错信息，否则继续向下进行下一次的重试。

在进行下一次重试之前，必须先恢复磁盘驱动的状态(例如磁头的位置等)，这一过程通过调用BIOS的 INT 13,0 来实现，因此我们需要设置AH,DL寄存器的值为0，然后使用INT 0x13来调用例程。最后使用一条JMP retry语句跳转到开头。

next

next块的作用是为读取下一个块做准备工作，包括设置柱面，扇区数，以及磁头等。在讲解代码之前，我们先梳理一下这段代码的读取逻辑，在读取十个柱面时，我们是一个柱面一个柱面的读取，读取每个柱面时，又是分磁头为0和磁头为1来读取。根据书中叙述的，一个磁盘有80个柱面，2个磁头，每个柱面18个扇区，每个扇区512B。因此大体上，我们读取时，先从第一扇区读到第十八扇区，然后切换磁头，再从第一扇区读到第十八扇区，然后切换柱面，重复上述步骤。

在next中，我们还需要对内存加载地址进行修改，由于每次都是读取一个扇区，512B，即0x0200大小的内容，因此[ES:BX]也应该增加这么多，不过介于该内存地址的计算方法是ES×16+BX，所以每次只需ES的值加0x0020。这里由于不能直接操作ES寄存器，所以要用AX做一下跳板。

接下来的逻辑就比较简单了，控制流到达next,说明上一个扇区已经读取成功了，增加CL的值，指明下一个要读取的扇区数，并与18比较，如果小于等于18，即这个柱面的“这一面”还没读完，那说明还可以继续用当前这个磁头读取，就直接跳转到readloop, 否则我们需要修改DH寄存器的值表示读取该柱面的另一面，同时不要忘记重置扇区数为1(MOV CL, 1)

给DH加1后，值可能变为2,这说明该柱面读取完成了，接下来，递增CH的值表示柱面数，并与设定的CYLS比较，看看是否达到了指定读取的柱面数，如果达到了，就会向下流入fin, 否则就设定磁头为0号，继续读取(扇区数之前就已经设定好了)。

值得注意的是，这里有一条新指令EQU, 它将一个标识符设置为一个定值，如上面的CYLS。

进入32位模式并导入C语言

说实话，我并不明白作者所谓的进入32位模式是什么意思，在整个代码中也没见到有明显的特权指令使得CPU从16位模式进入32位模式，这里就暂时不深究了。我就权当以为32位模式就是可以使用EAX,EBX等32位寄存器的模式。

这一部分的重点是：加载操作系统。在前面，我们已经完成了ipl程序，但它加载的磁盘并没有什么内容，而且在完成加载后只是一直空转，没有执行别的东西，这一部分，我们将完成一个简单的操作系统，并让它真正运行起来。

明确操作系统加载的位置

为了使操作系统能真正地运行起来，我们必须知道操作系统内容被加载到了哪个内存位置，之后在ipl中直接使用一条JMP跳转到指定内存位置即可。

在前面，我们说过，加载磁盘内容时，第二个扇区对齐内存0x8200的位置，后面的扇区顺着放置即可。因此，只需要知道操作系统在磁盘中的位置即可。

C程序编译过程

因为这本书使用的是作者设计的编译器，所以一切都要按照他的想法来。在这里，想要将C程序转为机器代码并写入磁盘，还有一系列的步骤：

  1. cc1.exe : bootpack.c -> bootpack.gas, 将C程序编译为gas汇编语言
  2. gas2nask.exe: bootpack.gas -> bootpack.nas, 这一步是将gas汇编语言转为我们一直使用的nas汇编语言
  3. nask.exe: bootpack.nas -> bootpack.obj, 这一步就是我们一直使用的编译过程，将nas汇编转为机器指令，但是这里的机器指令还未经过链接和重定向等处理。
  4. obi2bim.exe: 多个obj文件 -> bootpack.bim，这一步完成的是链接工作，将多个obj文件链接成为一个完整的bim文件(binary image file), 这里的bim文件是作者自己设计的一种格式
  5. bim2hrb.exe: bootpack.bim -> bootpack.hrb, 将上一步得到的bim文件转换为hrb文件，这里作者也没有提供过多的解释，可以理解为对文件格式进行了改进，增加了文件头等内容。

完成bootpack.hrb之后，就可以将bootpack.hrb与asmhead.bin放置在一起，形成haribote.sys文件了。这就是初步的操作系统内核文件，而要得到镜像文件，还需要ipl.bin。
