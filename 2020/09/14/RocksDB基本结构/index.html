RocksDB基本结构

前言

RocksDB是一种快速存储系统，改进于 LevelDB 并在一些地方做了新的优化和支持，例如：支持多线程读写，支持多线程的Compaction操作等等。本文将对RocksDB整体的构造布局做一些简单的介绍。

RocksDB基本结构

RocksDB的基本结构可以直接用下图进行展示：

上图按照硬件的存储层次将 RocksDB 的结构分为内存和磁盘上两大部分。内存部分主要由memtable 和 immutable memtable 组成。磁盘中的部分主要由多级文件结构组成。

上图还展示了 RocksDB 读写时操作，当RocksDB读数据时，会按照memtable-> immutable memtable -> block cache -> SST file(L0 -> L1 -> …) 这样的顺序进行查找。当写数据时，会先同时写入WAL 和 memtable, 前者的作用是保障一致性，后者则是作为存储系统的写缓存。memtable空间满后会转换为immutable memtable并磁盘刷新。

在逻辑上，LSM-Tree是RocksDB 的存储主体，下面介绍LSM-Tree。

LSM-Tree

LSM-Tree是RocksDB的基本组成部分之一，其主要作用是通过批量的存储技术来减少随机写导致的高昂的写代价。我们都知道，对于磁盘等存储系统，其读写时间有相当一部分都花费在寻道和旋转上。对于随机读写，会导致寻道和旋转频繁发生，从而降低读写性能。好的解决办法是将写内容先保存在内存中，因为内存的随机读写性能高，然后奖随机写转换为顺序写，利用磁盘的顺序写来提升写性能。

LSM-Tree就是通过批量存储的方式规避了随机写的问题，LSM的全称是Log-Structured Merge Tree, 在 RocksDB中，这是一种层次性的数据结构，类似于一棵树。当要写入数据时，RocksDB会先将数据写入到内存中一部分区域，当该区域中数据越来越大直到达到一个阈值时，就会生成一个称为 immutable memtable 的文件(意为不可修改的)并且 flush 到磁盘上(称为刷盘)，形成SST (sorted string table)文件。但是刷盘并不是随意地刷新到某个位置，正如上面所说，LSM是一种多层次的结构，此时刷盘会刷新到内存下磁盘的”第一层”。处于磁盘中的树会 周期性 的触发 merge 操作，这是一种将小的“树”合并为大“树”的操作，当该层合并后导致容量到达一个阈值时，就又会进行刷盘，刷新到更下面的层。该过程可以用下图简单地表示出来：

在上图中，处于memory的C0层满了之后(当然并不是以树的数据结构组织的)，就被刷新到C1层，成为一棵子树，它会和原先就驻留在C1层的其他子树一起形成一棵更大的“树”，这个过程就是合并(Merge), 但合并不是简单地将文件拼接在一起，而是一个更加复杂的过程。但SST文件本身内部有序的组织方式在一定程度上减少了Merge的复杂度。

当读数据的时候，RocksDB会先从内存中的 memtable 开始找，找不到就到 immutable memtable 中找，然后到磁盘的 L0, L1,… 中找，这个过程被称为读放大。RocksDB还有一个被称为manifest 的日志文件，它记录了 SST 的修改日志。

LSM-Tree 全称为 Log-Structured Merge Tree , 但它并不是一棵树，甚至不是一个具体的数据结构，而是一种存储设计上的概念，例如，在LSM-Tree的memtable部分，RocksDB采用的是SkipList 这样的数据结构来存储；而在磁盘部分，RocksDB则采用有序的文件集形式组织数据，而不是传统的B+树的形式。这些文件集的层次体现在写入时间或合并时间上，而并不体现为在磁盘上的物理组织方式。除此之外，LSM-Tree还具有以下几个特点：

  1. 写采用追加写的方式。当写入数据时，都是将数据写入一个或几个新的文件，采用顺序写入的方式并且不允许修改其他的文件。这样的一个问题就是当修改数据时，就会在数据库中产生一个键值对的多个副本，造成空间的浪费。
  2. 数据按照key排序。LSM-Tree通过Merge操作合并不同的SST文件，通过事先对数据采用有序组织的方式可以使得合并更加高效。同时使用有序组织也可以利用 二分查找 更高效地找到一个key在 SST 文件中的位置。
  3. 内部的Garbage Collect。上面已经提到了，追加写的方式容易导致一个数据的多个副本同时存在，造成空间的冗余，而RocksDB内部提供了Compaction操作作为一种垃圾收集的机制。包括对数据进行删除，定期淘汰一些过期的数据，同一个key的value合并等操作。

从上面的叙述不难看出，LSM-Tree实际上是牺牲了读性能(多次读取memtable, SST文件)，而提升了写的性能(将随机写转换为批量的顺序写)。原因在于读性能还能通过软件进行优化，而写性能更多地依赖于磁盘的IO。
