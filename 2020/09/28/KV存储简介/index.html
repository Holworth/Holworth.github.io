KV存储简介

NoSQL

NoSQL的含义是Not Only SQL， 是对不同于传统的关系型数据库的数据库管理系统的统称。

NoSQL一词最早出现于1998年，来源于Carlo Strozzi开发的一个轻量，开源的，不提供SQL功能的关系型数据库。该词在2009年于亚特兰大矩形的no:sql讨论会上被重新提出，被定义为“非关联性的”数据库，NoSQL更强调键值存储和面向文档数据库的优点。

NoSQL出现并在今年迅速发展的一个原因是进入互联网时代后，每天都将产生大量数据，以及与数据相关的查询，更新等操作。而传统的关系型数据库在高并发的使用情况下性能很低，而诸如Key Value Storage之类的NoSQL数据库则在这方面有着更加强劲的表现，具有更好的性能，可用性和可伸缩性。但NoSQL也存在一些缺点，例如它无法提供关系型数据库那样的ACID属性，只能提供相对弱一点的最终一致性模型，同时也无法提供更多的操作等等。

KV Store

KV Store指的是 Key Value Storage, 即键值存储系统。KV store是NoSQL运动的一部分，但实际上早在关系型数据库出现之前，KV存储的想法就已经得到了一定的应用，例如：使用hash表对数据进行索引。严格来说，所有的索引结构都可以看作是一种KV存储的体现，不同的是，早期我们通常只在内存中做这种事，而当现在KV真正成为一种存储系统，我们通常指的是将它应用于大规模存储中。Google发表的BigTable等一系列论文使得KV store逐渐走进了人们的视野之中。

KV存储，正如它听上去那样，是指使用<Key, Value>形式存储数据的一种形式。如果将每个key和 value的对作为一条记录，那么没有任何两条有效记录的key是相同的(我们使用有效是因为可能存在过期副本)。

KV存储和传统的关系型数据库的存储有很大的不同。关系型数据库中的基本数据单元也是一条“记录”， 但它通常是被预定义好的具有固定域的数据结构。例如，当我们想实现一个统计班上信息的数据库时，我们预先定义每条记录代表一位同学，这条记录的域包括：姓名，性别，年龄，家庭住址等等。在这种情况下，由多条记录组成的“表” 通常会表现为如下形式：

id 姓名 性别 年龄 家庭住址
4123 小明 男 12 xxxx
192 小红 女 11 xxxx
9023 小刚 男 12 xxxx

如果我们想使用关系型数据库来实现它，我们可能需要多个表：Address, Age表，然后将一个student id中的address id关联到Address表上，诸如此类的。而在KV存储中，我们不需要表，也不需要为表设计关系，上面的一条记录就是value的值，通过索引student id就能得到一条记录的全部信息。

此外，对于KV存储而言，也不一定有如此规则的数据域。在KV存储中，每一条记录的value域通常被认为是无规则的字节流(当然也有一些固定域，例如在磁盘的偏移位置，但总的来说，并不如RDB(relation database)那样规则)。它们的每条记录具有相当的灵活性，是非常自由的数据模式，如下所示：

key value
K1 AAA, BBB, CCC
K2 AAA, BBB
K3 AAA,2,01/01/2015
K5 3, ZZZ, 5623

KV存储这样的存储特性带来了一些局限性，例如：不能根据value域的值来提取对应的key， 而在关系型数据库中，则能通过“提取年龄为12的记录”对应的SQL语句提取表中所有符合条件的记录，换言之，KV存储系统缺乏过滤功能。这意味着KV存储系统在所能提供的操作上会受到很大局限。

不过KV存储模型也有很多的好处，一是KV store对于高并发的需求支持得很好，二是KV store的扩展性较强。当存储数据量变大时，通过增加机器就能达到较好的效果。
