LSM-Tree Compaction简介

前言

在前面已经提到，RocksDB采用LSM-Tree作为自己存储系统的主体部分, 从而以牺牲一部分读性能为代价提升了写性能。LSM-Tree的一个不足之处在于追加写的方式(Put, Update, Delete都是采用追加写的方式，Delete是写入一个删除标志)会导致存储系统中同一个key存在多个不同版本的value, 从而浪费空间。因此LSM-Tree的一个核心问题就是如何进行垃圾收集。LSM-Tree使用一种称为 Compaction 的机制进行垃圾收集，包括清除过期记录，合并具有相同key的键值对，收集闲置的空间等等。

Compaction算法

本文先对LSM-Tree的Compaction算法做一些介绍，以便以后更好地分析RocksDB中使用的Compaction策略。

LSM-Tree的Compaction算法中，为人熟知的两种分别是 Size/Tiered Compaction 和 Leveled Compaction 。两种算法的核心做法都是将较低层的SST文件合并为较高层的SST文件，但是在合并时，具体的操作策略却有所不同，例如：Tiered Compaction倾向于将低一层的所有SST文件合并为一个新的SST文件后再加入到较高层，而Leveled Compaction在每次合并时只会将底层的一个SST文件选出来，与高层的文件合并。不同的合并策略导致了两种算法不同的行为，进一步导致了两种算法不同的性能表现。

这里引用参考文献[1]中的一幅图像来展示两种Compaction的不同：

上图先展示了两种Compaction策略的极端情形，左上角的log表示根本不进行Compaction，每次都写入到磁盘上，这样显然大大提升了写性能，因为写的时候根本不需要关心Compaction等操作，每次都是直接追加，但是坏处在于查询效率极其低下，原因是存储系统中可能存在一个key的多个value副本,只能进行线性查找；第二种极端情形就是右下角的 sorted array ， 它表示的含义是内存内容每次一刷新到盘上，就执行Compaction, 这样的好处当然是几乎时刻都能保持磁盘上的数据是最新且一致的，但是过度的Compaction会严重导致更新效率的下降。

在log和sorted array连成的这条线上，还分布着几个典型数据库，可以看出RocksDB，LevelDB更加接近于sorted array, 而Cassandra更加接近于log。几种数据库都采用了LSM-Tree的模式，但RocksDB和LevelDB采用了Leveled Compaction的策略，而Cassandra采用的是Tiered Compaction的策略。

Tiered Compaction

Compaction策略

Tiered Compaction可以说是最早出现的一种Compaction策略。它的要点有以下几个：

  1. SST文件的大小限制可变，通常来说，随着层次的增加，一个SST文件的大小会越来越大
  2. LSM-Tree的每一层对SST文件数目进行限制，文件数目超过该限制就进行合并，且各层的数目限制相同
  3. 合并时第一层的 所有文件 合并为一个大的SST文件，放入到下一层。

一个经典展示Tiered Compaction合并过程的图如下所示：

从上图可以看出，假设每一层SST文件的数目限制默认为4，当该层数目达到4时，就会触发合并，将该层的4个SST文件合并为一个更大的SST文件，然后写入到下一层。层次越高，单个SST文件的容量也就越大。

性能分析

Tiered Compaction的一个严重的性能问题就是其空间放大问题。所谓空间放大(Space Amplification), 就是指：在存储系统中，数据实际占有的空间比数据实际大小更大的情形 。Tiered Compaction的空间放大来源于它的Compaction执行。

当对一个Tier的所有SST文件进行Compaction时，原先的SST文件在Compaction完成之前是不会删除的，这就导致了在Compaction过程中，实际上存在SST文件的两个备份：原SST文件，合并中的SST文件。在最坏的情况下，占用的空间可以达到原先的两倍。

参考文献[2]中给出了测试Tiered Compaction空间放大的过程：通过持续向Scylla数据库以每秒3MB的速度写入数据，共持续3000s, 最后的结果应该是9GB左右，整个过程中磁盘使用量的情况如下所示：

在整个过程中，我们可以看见许多毛刺状的突起，这就是短视的空间放大。尤其是在大约2000s左右的时候，有一个巨大的突起。根据计算，此时实际数据量应该为6GB, 但图中显示却达到了12GB, 接近两倍。

第二个实验是重复写入一个400万条数据的集合，大小约为1.2GB, 通过重复写入15次来模拟数据更新，时间和磁盘占用的曲线图如下所示：

上述结果显示出来的空间放大极为严重，最多可达到9.3GB, 几乎是实际数据量的8倍。其原因在于尽管能通过Compaction消除本层的多个重复SST, 但不断进入的重复数据仍然会驻留在较底层，导致多个重复的key。

Leveled Compaction

Compaction策略

相比Tiered Compaction, Leveled Compaction有以下几个特点：

  1. 每个SST文件大小不变
  2. 每一层通过也通过限制SST文件的数目来触发合并(和 Tiered Compaction相同)，但是每一层的数目限制不同，通常是随层次增加而指数增长
  3. 每一层都维护唯一一个 run

所谓 run 就是指一堆数据的集合：要么是一个内部有序的SST文件；要么就是多个内部有序的SST文件，但是这些SST文件的key的范围互不重叠。

Leveled Compaction通过维护唯一一个run可以达到快速的查询的目的：如果该层的run是一个SST文件，那么直接采用二分查找即可；如果该层的run是多个范围互不重叠的SST文件，那么由于其范围互不重叠，可以找出包含目标key的SST文件(一定是唯一的），然后在该文件内部采用二分查找找到目标key。

合并过程

下图很好地展示了Leveled Compaction的合并过程：

可以看出，不同于Tiered Compaction中单个SST文件容量的不断增长，在Leveled Compaction中，每个SST文件的大小都是相同的，每一层只是存储的SST文件的数量有所增加。减少SST文件的容量的好处在于在一个SST文件内部使用二分查找所占用的时间也会更少。

更加具体的Leveled Compaction的合并过程可以参考[3]。

Leveled Compaction和Tiered Compaction的另一个显著不同，正如在[3]中提到的那样，在于它的合并过程并不是将低层次的所有SST文件合并为一个新的SST文件然后刷新到高一层中，而是选取低层次的一个SST文件，将它与高层次中与其key范围有交叠的那些SST文件合并。这个过程能保持Leveled Compaction中要求的“每一层维护唯一一个run”。因为当一个SST和高一层与它有交叠的SST合并时，这些SST在序的关系上来看一定是相邻的，从而可以看作是将该SST中的新数据更新到高一层的多个SST中，这个过程并不会改变这些相邻SST的边界关系。

性能分析

Leveled Compaction的一个严重问题就是其写放大，所谓写放大是指在向磁盘或SSD写入数据时，实际写入的数据量是实际数据量的数倍。 Leveled Compaction写放大的原因在于其每次Compaction都会将低层次的一个SST文件与高层次中和它的key范围有重叠的所有SST文件合并，这个重叠的SST文件的数目可能会非常巨大，从而导致重写的数目非常巨大。尽管Tiered Compation也有写放大，但是Leveled Compaction的写放大更加严重。

不过实际应用中，Leveled Compaction的写放大并没有那么严重，一个合理的解释源于程序的时间局部性：一个新写入的数据很有可能在短时间内被再次修改。这部分数据通常存储在内存或L0中，它们很少向更高层次移动，从而在Compaction时，包含它们的SST和高一层次的SST的交叠较少，从而减少了写放大。

Leveled Compaction 和 Tiered Compaction的比较

上面的内容已经包含了部分两种方式的比较，下面以[1]一个例子来说明：

上图可以看出两种Compaction方式的一些不同，首先，Tiered的方式中，SST 4到达L1之后，触发合并，和1,3,7合并成为一个更大的SST文件，刷新到L2，此时L2中实际上是不止一个run的，因为2,5,6,8这个SST和1,3,4,7这个SST的key范围是有重叠的。相比之下，Leveled的方式最后将2,5,6,8也进行了合并，形成了唯一一个run。

上述两种不同的方式导致了在读效率上的不同。对于Tiered 而言，同一层中不同的run之间的key范围有重叠，这意味着在读的时候可能需要读取多个SST文件；而Leveled Compaction则不会有这个问题。因此Leveled Compaction有更好的读性能

在写性能上，Leveled Compaction的更新需要更加频繁，具有严重的写放大问题，而频繁的写会严重损耗SSD和磁盘的寿命。
